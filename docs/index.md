## Custom Project

### Dataset

The producer `kafka_producer_sailors.py` uses the unmodified `data/sales.csv`
dataset.

This dataset contains e-commerce purchase and order records.

Each row represents a single transaction event sent into Kafka.

Fields in the dataset include:

- `order_id`: unique identifier for the transaction
- `datetime`: timestamp for the purchase event
- `region_id`: geographic region
- `currency_code`: currency used
- `product_id`: purchased product ID
- `unit_price`: price per unit
- `quantity`: number of units purchased
- `is_online`: (T/F) whether the purchase was online
- `customer_id`: unique customer identifier
- `is_new_customer`: whether the customer is new
- `device_type`: device used
- `payment_method`: payment type
- `referral_source`: traffic source
- `discount_code`: optional discount code
- `customer_note`: optional customer comment

### Kafka Messages

- The producer sends each row from `data/sales.csv` to Kafka topic
  `streaming-02-kafka-sailors`.

- The producer uses `region_id` as the message key.

- This splits messages by region in Kafka.

- Message fields remain unchanged.

### Consumer Processing

The consumer file `kafka_consumer_p5.py` receives messages from the
`data/sales.csv` dataset.

Each message represents one sales transaction.

It reads all available messages on each run.

Consumption continues until either the maximum of 1000 messages is reached
or no message is received for 10 seconds.

Each received message is logged as a full dictionary.

The consumer also tracks message frequency per Kafka key in real time.

All output is written to `data/output/consumed_sales_p5.csv`.

No fields are filtered.

### Experiments

#### Phase 4: Small Technical Modification

- The `KAFKA_TOPIC` was changed from `streaming-02-kafka-case`
  to `streaming-02-kafka-sailors`.
- `kafka_producer_case.py` was copied to `kafka_producer_sailors.py`.
- `kafka_consumer_case.py` was copied to `kafka_consumer_sailors.py`.
- The output file was changed to `consumed_sales_sailors.csv`.

#### Phase 5: Apply the Skills

- `kafka_consumer_sailors.py` was copied to `kafka_consumer_p5.py`.
- The output file was changed to `consumed_sales_p5.csv`.
- A message count by Kafka key was added in `kafka_consumer_p5.py`.
- The consumer was extended to track message counts per Kafka key.
- This enabled basic stream analytics without changing producer logic.

### Results

The producer sent 3 messages to the Kafka topic.

The consumer read all 3 messages successfully.

All messages used the Kafka key `NULL`.

### Interpretation

The Kafka workflow showed data flowing from producer to topic to consumer.

Each message was processed one at a time in order.

Kafka delivered records while the consumer maintained simple state.

No external storage was required for tracking counts or progress.

The stream showed customer activity patterns such as product usage,
regional distribution, and referral sources.
